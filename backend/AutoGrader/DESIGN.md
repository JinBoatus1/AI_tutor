# AutoGrader 设计草案

## 1. 目标

构建一个可模块化扩展的试卷评分系统，能够：

- 接收多种输入格式：PDF 文件，或者属于同一张卷子的多张图片。
- 接收同样格式的标准答案。
- 使用文本 prompt 作为评分指令。
- 支持多份试卷并发评分。
- 第一版只返回分数和状态码。

本文档用于在接入 API 路由前，先明确当前设计方向。

## 2. 设计原则

1. 所有外部输入先统一为一种内部表示。
2. API 层保持轻量，不把评分逻辑直接写进路由函数。
3. 预处理、评分、聚合、结果持久化要解耦。
4. 预留未来多模型联评能力，但不改变外部协议。
5. 批量评分优先采用任务式异步执行。

## 3. 当前范围

### 范围内

- 学生答案输入支持 PDF 或图片组。
- 标准答案输入支持 PDF 或图片组。
- 使用文本 prompt 作为评分依据。
- 输出分数。
- 输出状态码。
- 支持多份试卷并发执行。

### v1 暂不包含

- 详细 rubric 拆分。
- 按题目分项计分。
- 人工复核界面。
- 完整 OCR 流程实现。
- 最终 API 路由接入。
- 存储后端定型。

## 4. 输入标准化

系统应把所有外部输入转换为统一的文档抽象。

### DocumentBundle

DocumentBundle 表示一份逻辑上的试卷或标准答案。

建议字段：

- `bundle_id`：文档组的唯一标识。
- `kind`：`student_paper` 或 `answer_key`。
- `sources`：源文件列表。
- `metadata`：可选附加信息，例如科目、考试名称、年级。

建议约定：

- 一份完整 PDF 作为一个原始 source 保留，不在存储层强制拆成单页。
- 同一张卷子的多张图片作为一个 bundle 内的多个 source 统一管理。
- 是否拆页、是否合并成 PDF、是否直接传给 agent，属于执行层策略，不属于存储层规范。

### SourceItem

SourceItem 表示一个具体输入来源。

建议字段：

- `source_type`：`pdf` 或 `image`。
- `uri`：文件路径、上传 key 或存储引用。
- `mime_type`：例如 `application/pdf` 或 `image/png`。
- `page_hint`：图片组的页序提示，可选。
- `checksum`：用于去重和缓存复用的哈希值，可选。

### 标准化规则

- PDF 输入在保存时应保留为完整文件对象。
- 图片输入应按卷子聚合成 list，保留页序和来源信息。
- 是否在进入 agent 前拆页、渲染或合并，交给执行层根据模型能力动态决定。
- 如果后续加入 OCR 或视觉解析，也应优先消费原始 bundle，再按需要生成中间标准化页面。

### 传给 Agent 的策略

- 如果目标模型原生支持 PDF，可直接传完整 PDF。
- 如果目标模型更适合图片输入，可将 PDF 转成页面图片后传入。
- 多张图片可以直接作为 list 传入，也可以按需要先合并为 PDF 再传入。
- 具体策略由 evaluator 或 adapter 决定，不应写死在文档存储结构里。

## 5. 输出约定

第一版结果应尽量保持最小化。

### AutoGradeResult

建议字段：

- `score`：数值分，通常范围为 0 到 100。
- `status_code`：机器可读的结果码。
- `message`：可选的人类可读说明。

### 状态码建议

- `OK`
- `ACCEPTED`
- `PENDING`
- `RUNNING`
- `DONE`
- `PARTIAL_DONE`
- `INVALID_INPUT`
- `UNSUPPORTED_FORMAT`
- `FILE_NOT_FOUND`
- `PREPROCESS_FAILED`
- `TIMEOUT`
- `EVAL_ERROR`
- `INTERNAL_ERROR`

## 6. API 方向

更推荐使用 job-based 接口，而不是单次阻塞式评分接口。

### 建议接口

#### `POST /api/autograder/jobs`
创建一个批量评分任务。

请求字段：

- `prompt`：评分说明文本。
- `items`：评分任务列表。

每个 item 应包含：

- `paper_id`
- `student_bundle`
- `answer_bundle`

返回字段：

- `job_id`
- `accepted_count`
- `status_code`

#### `GET /api/autograder/jobs/{job_id}`
查询任务状态。

返回字段：

- `job_id`
- `status_code`
- `total`
- `completed`

#### `GET /api/autograder/jobs/{job_id}/results`
获取最终结果。

返回字段：

- `paper_id`
- `score`
- `status_code`

## 7. 内部架构

### 第 1 层：API / 提交层

职责：

- 接收请求。
- 校验输入结构。
- 把请求转换成内部任务对象。
- 返回 job_id 或即时校验错误。

### 第 2 层：调度器

职责：

- 维护任务队列。
- 控制并发数。
- 将任务分发给 worker。
- 跟踪任务状态。

### 第 3 层：Worker 池

职责：

- 一次处理一个评分任务。
- 执行预处理和评分。
- 处理超时和失败。
- 把结果写回结果存储。

### 第 4 层：评估器

职责：

- 从不同角度评估试卷。
- 支持多个模型。
- 返回结构化中间结果。

### 第 5 层：聚合器

职责：

- 合并多个评估器的输出。
- 计算最终分数。
- 处理分歧或降级情况。

## 8. 多模型联评方案

系统应支持多个评估代理并行运行。

### 评估流程

1. 输入标准化。
2. 创建评估任务。
3. 将任务分发给多个模型适配器。
4. 收集模型输出。
5. 聚合分数。
6. 返回最终结果。

### 建议评估维度

- 正确性
- 完整性
- 推理质量
- 表达质量

每个维度都可以实现为一个独立评估器，或者作为共享模型适配器的 prompt 配置。

## 9. 并发方案

这个模块需要能安全处理大量同时评分请求。

### 推荐策略

- 对 I/O 密集的模型调用使用异步调度。
- 使用 semaphore 或有界队列控制并发上限。
- 每个任务保持状态隔离。
- 避免共享可变全局状态。
- 只有在本地 CPU 预处理较重时，才引入 worker 进程。

### 默认限制建议

这些值只是暂定，后续再调优：

- 最大并发任务数：8 到 12
- 每个任务内最大并发模型调用数：2 到 4
- 单次模型调用超时：可配置
- 单份试卷整体超时：可配置

## 10. 存储与缓存

后续版本可能需要持久化存储：

- 任务队列状态
- 结果记录
- 输入文件引用
- 原始 PDF 文件
- 原始图片列表
- 预处理后的页面缓存
- 基于 checksum 的去重

当前还没有固定存储后端。设计上应保持与存储实现解耦。

## 11. 建议模块结构

当前文件夹后续可演化为：

- `models.py`：请求、结果和公共 schema。
- `grader.py`：评分器抽象基类。
- `service.py`：服务定位器或工厂。
- `scheduler.py`：任务队列和分发逻辑。
- `orchestrator.py`：评分流程编排器。
- `evaluators/`：各个独立评估器或模型适配器。
- `aggregator.py`：最终分数聚合。
- `preprocess.py`：图片/PDF 标准化。
- `store.py`：任务与结果持久化。

## 12. 待确认问题

以下问题还需要团队讨论：

1. API 是否应立即返回 job_id，还是在小任务时阻塞到评分完成？
2. 输入是上传文件、文件引用，还是两者都支持？
3. OCR 是必需的预处理步骤，还是仅作为兜底？
4. 分数范围和归一化规则具体怎么定？
5. v1 是否需要分维度分数，还是只保留总分？
6. 批量输出里失败卷子要如何表示？
7. 第一版最多支持多大的批量规模？
8. 首批需要支持哪些模型适配器？

## 13. 下一步建议

先把输入协议定下来，重点包括：

- 一份试卷 bundle 如何表示。
- 一份标准答案 bundle 如何表示。
- 批量任务如何提交。
- 分数和状态码如何返回。

当这个协议稳定之后，调度层和评估层就可以直接实现，不必再重做 API 设计。
